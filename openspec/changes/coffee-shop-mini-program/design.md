## Context

单店咖啡奶茶店需要一个微信小程序实现顾客自助点单、取餐功能。商家希望通过独立网页管理商品和订单。

**现状**：
- 顾客到店排队点单，高峰期等待时间长
- 无现有系统，需要全新开发
- 菜单约14个SKU，无规格选项

**约束**：
- 后端：Node.js + SQLite
- 前端小程序：Taro + React
- 商家管理：独立网页
- 开发方式：本地先行

## Goals / Non-Goals

**Goals:**
- 顾客可在小程序浏览菜单并完成下单支付
- 顾客获得取餐号，按时到店凭号取餐
- 商家可在网页端管理商品（上下架、价格）和查看/更新订单状态
- 接入微信支付完成收款

**Non-Goals:**
- 不支持外卖配送
- 不支持会员系统/储值卡
- 不与现有POS系统集成
- 暂不部署到云服务器（本地开发测试）

## Decisions

### 1. 小程序框架：Taro + React

**决定**：使用 Taro 3.x + React 开发微信小程序。

| 方案 | 优点 | 缺点 |
|------|------|------|
| 原生开发 | 性能最优、官方支持 | 代码不可复用、学习成本高 |
| **Taro + React** | **一套代码多端复用、社区活跃** | 性能略低于原生 |
| UniApp + Vue | 一套代码多端复用 | 社区相对较小 |

### 2. 后端框架：Express + node:sqlite

**决定**：使用 Express.js + Node.js 24 内置的 `node:sqlite` 模块。

- Express：成熟稳定，文档完善
- node:sqlite：Node.js 24 内置，无需原生编译，性能优秀

### 3. 商家网页：React 单页应用

**决定**：使用 React + Vite 构建独立商家管理网页。

- 与小程序共用同一套后端API
- 通过路由区分顾客端（小程��）和商家端（网页）

### 5. API 设计

| 方法 | 路径 | 描述 |
|------|------|------|
| GET | /api/products | 获取商品列表 |
| GET | /api/products/:id | 获取单个商品 |
| POST | /api/orders | 创建订单 |
| GET | /api/orders/:id | 获取订单详情 |
| PATCH | /api/orders/:id/status | 更新订单状态 |
| GET | /api/orders | 获取订单列表（商家） |
| GET | /api/wechat/pay params | 获取微信支付参数 |
| POST | /api/wechat/callback | 微信支付回调 |

### 6. 取餐号生成规则

按日顺序编号：`日期 + 序号`（如 `20260628-001`）

- 每日从001开始
- 通过 `daily_counter` 表管理当日最大序号

## Risks / Trade-offs

| 风险 | 说明 | 缓解措施 |
|------|------|----------|
| 微信支付申请被拒 | 微信支付商户号申请有门槛 | 申请商户号前准备营业执照等材料；可先开发支付流程，暂不接入真实支付 |
| SQLite 并发写入 | 高峰期多顾客同时下单 | SQLite 写入锁粒度细，高峰期单店场景可满足；未来可升级 PostgreSQL |
| 小程序审核被拒 | 功能或命名不符合规范 | 提前了解微信小程序审核要点；店铺类目选择正确 |
| 商家网页安全 | 商家端需要认证机制 | 开发初期先做简单认证；后续增加 token 机制 |
